# Argo Energy Solutions — Claude Code Context

## Project Overview

Energy monitoring platform that ingests data from Eniscope sensors into a Neon
(PostgreSQL) database. Reads come in via the Eniscope API; analytics and reports
are generated by Python scripts in `backend/python_scripts/`.

---

## Neon Database — MCP Connection

The `neon` MCP server is configured in `.claude/mcp.json`. It connects to the
Neon PostgreSQL database using `DATABASE_URL` from `.env`.

**Prerequisites:**
1. Copy `.env.example` → `.env` and fill in your `DATABASE_URL`.
2. `DATABASE_URL` format: `postgresql://user:password@host/dbname?sslmode=require`

The MCP server exposes `query` and `schema` tools so you can ask questions about
your data directly in Claude Code.

## Database Schema

### Governance Rule
**Always query through Layer 3 Business Views.** Never query raw tables directly
in analytics work — use the views below to get clean, enriched data.

---

### Raw Tables (source of truth — ingest only)

| Table | Description |
|---|---|
| `readings` | Raw energy readings from Eniscope |
| `channels` | Measurement channels (one per circuit/sensor) |
| `devices` | Physical Eniscope hardware devices |
| `organizations` | Customer organisations / sites |
| `sites` | Active monitored locations (governance registry) |
| `meters` | Device-level meter metadata from /meters API endpoint |
| `virtual_meters` | Synthetic meters defined as weighted sums of channels |
| `virtual_meter_map` | Channel weights for each virtual meter |

#### `readings` columns
`id`, `channel_id`, `timestamp` (TIMESTAMPTZ), `energy_kwh`, `power_kw`,
`voltage_v`, `current_a`, `power_factor`, `reactive_power_kvar`,
`temperature_c`, `relative_humidity`, `frequency_hz`, `neutral_current_a`,
`thd_current`, `apparent_power_va`, `cost`,
phase fields: `voltage_v1/2/3`, `current_a1/2/3`, `power_w1/2/3`,
`power_factor_1/2/3`, `energy_wh1/2/3`, `created_at`

#### `channels` columns
`channel_id`, `channel_name`, `organization_id`, `device_id`,
`channel_type`, `unit`, `meter_id`, `created_at`, `updated_at`

#### `sites` columns
`site_id` (= Eniscope `organization_id`), `site_name`, `is_active`,
`wcds_only`, `resolution` (seconds), `timezone`, `notes`,
`created_at`, `updated_at`

### Layer 1 View

| View | Description |
|---|---|
| `v_clean_readings` | Raw readings filtered to non-null, non-zero energy & power; timestamp cast to TIMESTAMPTZ |

---

### Layer 2 Materialized View

| View | Description |
|---|---|
| `mv_hourly_usage` | Hourly aggregates: `meter_id`, `hour`, `avg_kw`, `total_kwh`, `reading_count`, electrical health fields. Unique index on `(meter_id, hour)` — supports `REFRESH CONCURRENTLY`. |

Refresh after each ingestion run:
```sql
REFRESH MATERIALIZED VIEW CONCURRENTLY mv_hourly_usage;
```

---

### Layer 3 Business Views (use these for analytics)

| View | Best for |
|---|---|
| `v_readings_enriched` | General analytics — readings with site, meter, device names |
| `v_meters` | Meter/channel list with site and device context |
| `v_sites` | Site list (alias for `organizations`) |
| `v_latest_readings` | Current status / dashboard — one row per meter |
| `v_readings_hourly` | Hourly trends (wraps `mv_hourly_usage`) |
| `v_readings_daily` | Daily aggregates: total energy, avg/peak power, voltage, power factor, electrical health |
| `v_readings_monthly` | Monthly aggregates for budgets and billing |
| `v_active_sites` | Sites with `is_active = true` |
| `v_data_health_monitor` | Data quality — `last_reading_at`, `minutes_since_last_reading`, `status` (HEALTHY / WARNING / CRITICAL) |
| `v_virtual_readings` | Aggregated readings for virtual/synthetic meters |

#### `v_readings_enriched` key columns
`id`, `meter_id`, `meter_name`, `site_id`, `site_name`,
`device_id`, `device_name`, `device_type`, `timestamp`,

## Common Query Patterns

### List all meters
```sql
SELECT meter_id, meter_name, site_name, channel_type
FROM v_meters
ORDER BY meter_name;
```

### Energy summary — last 7 days
```sql
SELECT
    meter_name,
    ROUND(SUM(energy_kwh)::numeric, 2) AS total_kwh,
    ROUND(AVG(power_kw)::numeric, 2)   AS avg_kw,
    ROUND(MAX(power_kw)::numeric, 2)   AS peak_kw
FROM v_readings_enriched
WHERE timestamp >= NOW() - INTERVAL '7 days'
GROUP BY meter_id, meter_name
ORDER BY total_kwh DESC;
```

### Daily trend for a specific meter
```sql
SELECT reading_date, total_energy_kwh, avg_power_kw, peak_power_kw
FROM v_readings_daily
WHERE meter_name ILIKE '%RTU-1%'
ORDER BY reading_date DESC
LIMIT 30;
```

### Current status of all meters
```sql
SELECT meter_name, site_name, timestamp, power_kw, voltage_v, power_factor
FROM v_latest_readings
ORDER BY meter_name;
```

### Data health check
```sql
SELECT channel_name, last_reading_at, minutes_since_last_reading, status
FROM v_data_health_monitor
ORDER BY status DESC, minutes_since_last_reading DESC NULLS FIRST;
```

### Monthly energy by meter
```sql
SELECT meter_name, reading_month, total_energy_kwh, avg_power_kw
FROM v_readings_monthly
ORDER BY meter_name, reading_month DESC;
```
